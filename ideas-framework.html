✨ treval · Ideas Framework

Observabilidad + evaluación para agentes de IA · CLI-first · OTEL nativo

1. ¿Qué debería tener treval?

Trazar, evaluar y depurar agentes de IA desde la terminal — minimizando fricción y dependencias externas. Basado en la tríada: decoradores + auto-instrumentación + OTEL.

Áreas core

  1. Tracing — capturar cada paso de un agente: tool calls, LLM calls, decisiones, errores
  2. Métricas — latencia, tokens, coste, count de tools por sesión
  3. Evaluación — LLM-as-judge, tests unitarios para agentes (treval test run)
  4. Dashboard local — ver traces desde terminal o navegador local (sin cloud)
  5. Comparación A/B (experimentos) — correr mismo dataset con 2 configs y comparar
  6. SDK ligero — que quepa en una tarde de leer el README, sin dependencias pesadas

2. Patrones de integración (de más simple a más robusto)

🎯 Decoradores

El más ergonómico. Añades @treval.agent, @treval.tool, @treval.trace sobre tu código y ya está trazado.

@treval.tool
def search_web(query: str) -> list[Result]:
    return tavily_search(query)

@treval.agent
class SupportAgent:
    @treval.trace
    def handle(self, msg: str) -> str:
        ...
⚡ Auto-instrumentación

Una línea de código, zero-change. Parchea las librerías conocidas (OpenAI, Anthropic, HTTP) para trazarlas automáticamente.

import treval
treval.instrument()  # ← y ya

from openai import OpenAI
client = OpenAI()  # ← todo lo que hagas con esto queda trazado
🔌 Wrapping de clientes

Envolver los clientes de OpenAI/Anthropic para inyectar tracing sin modificar el código del agente.

from openai import OpenAI
client = treval.wrap(OpenAI())  # ← wrapper transparente
🌐 Proxy / Gateway

Interceptar tráfico HTTP a nivel de red. Útil cuando no puedes tocar el código del agente (agentes de terceros, legacy).

# El agente apunta a http://localhost:4318/v1
# treval reenvía al proveedor real y traza todo
📡 Callbacks / Hooks

Patrón observer: el framework del agente (LangChain, CrewAI, etc.) llama a treval cuando ocurren eventos.

from treval.integrations.langchain import TrevalCallbackHandler
agent = create_react_agent(llm, tools, callbacks=[TrevalCallbackHandler()])
📊 OpenTelemetry nativo

Estándar abierto CNCF. Sin vendor lock-in — exportable a Langfuse, Grafana, Phoenix, o tu propio backend. La capa de transporte definitiva.

# Código agnóstico — el exportador decide el destino
from opentelemetry import trace
tracer = trace.get_tracer("treval", "0.1.0")

Recomendación

Combinar decoradores + auto-instrumentación + OTEL como base:

┌─────────────────────────────────────────────────────────┐ │ ⚡ Auto-instrumentación → captura lo genérico │ │ + │ │ 🎯 Decoradores → captura lo específico │ │ ↓ │ │ 📊 Todo en formato OTEL → cualquier backend │ └─────────────────────────────────────────────────────────┘

Auto-instrumentación captura LLMs, HTTP, librerías conocidas — sin tocar código.
Decoradores capturan tu lógica de negocio (tools, agentes propios) — con 1 línea.
OTEL garantiza datos portables a cualquier backend.

El wrapping de clientes y los hooks quedan como integraciones opcionales para frameworks concretos (LangChain, CrewAI). El proxy es para casos extremos de legacy.

3. Herramientas existentes — análisis detallado

🔮 Langfuse — 28.8k ⭐

La más popular del ecosistema open source. Traza agentes y LLMs con OTEL nativo, gestión de prompts, experiments y un dashboard web completo.

Qué hace:

  • Decorador @observe() genérico para cualquier función
  • Drop-in replacement del cliente OpenAI
  • OTEL nativo (OTLP)
  • Prompt management con versionado
  • Agent Graphs — visualización jerárquica de agentes
  • Experiments con datasets
  • 50+ integraciones

Por qué es relevante: Si treval es compatible con OTEL, cualquier usuario de Langfuse podría usar treval como CLI/local y mantener Langfuse para producción. Son complementarios, no rivales.

Qué NO hace bien (oportunidad para treval):

  • No es CLI-first — todo es dashboard web
  • Requiere infraestructura (Langfuse Cloud o self-host con Postgres + S3)
  • El decorador @observe() es genérico — no distingue entre tool, agent, operation
  • No tiene replay interactivo desde terminal

🛡️ AgentOps — ~5k ⭐

El modelo de datos más rico para agentes. Tiene 7 decoradores específicos con una jerarquía bien definida.

@session        → Sesión completa de usuario
  └── @agent    → Clase agente (ciclo de vida)
       ├── @operation / @task   → Unidad de trabajo
       │    ├── LLM (automático)
       │    └── @tool           → Tool call con coste
       └── @workflow            → Secuencia de operaciones
@guardrail      → Input/output de guardrails

Por qué es relevante: Es el que tiene el diseño más parecido a lo que necesita treval. La jerarquía de spans con tipo específico (SESSION → AGENT → OPERATION → TOOL) es el modelo correcto. AgentOps ha validado que esta estructura funciona en producción.

Qué NO hace bien (oportunidad para treval):

  • Requiere AgentOps Cloud (API key, cuenta) — no hay modo 100% local
  • Dashboard web, no CLI
  • No está basado en OTEL — usa su propio protocolo
  • No tiene evaluación/testing integrado (solo tracing)
  • No tiene SQLite local — todo va a su cloud

🦅 Phoenix / Arize — 10.1k ⭐

100% OTEL + OpenInference. Un equipo (Arize AI) manteniendo activamente el estándar OpenInference para atributos semánticos de LLMs.

Qué hace:

  • OTEL nativo vía OpenInference
  • Auto-instrumentación para frameworks (LlamaIndex, LangChain, DSPy, Vercel AI SDK)
  • Tracing, evaluación, prompt engineering, datasets & experiments
  • PXI — un agente built-in para debuggear traces desde la UI
  • Evaluación LLM-as-judge + custom evaluators
  • Python, TypeScript, Java

Por qué es relevante: Es la referencia de cómo hacer OTEL bien para LLMs. OpenInference es el estándar de facto para atributos semánticos. Si treval adopta OpenInference, es compatible con Phoenix desde el día 1.

Qué NO hace bien (oportunidad para treval):

  • No es CLI-first — todo el flujo está pensado para el dashboard web
  • Auto-instrumentación solo para frameworks grandes — no para código custom
  • No tiene decoradores específicos — todo es OTEL genérico
  • Requiere levantar un servidor Phoenix (docker o local)

🧪 DeepEval — 16.1k ⭐

Enfoque completamente diferente: testing como pytest. No es observabilidad en tiempo real — es evaluación offline/batch.

from deepeval import assert_test
from deepeval.metrics import GEval

def test_correctness():
    correctness_metric = GEval(
        name="Correctness",
        criteria="Determine if the actual output is correct...",
        threshold=0.5
    )
    test_case = LLMTestCase(
        input="¿Me duele la cabeza?",
        actual_output="...",
        expected_output="..."
    )
    assert_test(test_case, [correctness_metric])

# Se ejecuta con:
# deepeval test run test_example.py

Por qué es relevante: DeepEval ha demostrado que los developers quieren evaluar agentes como evalúan código — con un CLI, tests, CI/CD. 14+ métricas pre-built (GEval, Faithfulness, Hallucination, etc.). El approach de testing-first está probado.

Qué NO hace bien (oportunidad para treval):

  • No tiene tracing — solo evaluación offline
  • No tiene observabilidad en tiempo real
  • Métrica GEval requiere OpenAI sí o sí (no trae juez propio)
  • No tiene decoradores — todo es assert_test
  • No guarda traces, solo resultados de tests

📦 Otras herramientas consideradas

  • W&B Weave — OTEL-based, integración con Weights & Biases. Bueno para ML experiments. Demasiado atado al ecosistema W&B.

4. Diferenciadores de treval (el gap real)

🎯 CLI-first — ver traces sin abrir un navegador

Problema: Langfuse, AgentOps y Phoenix obligan a abrir un dashboard web. Cuando estás desarrollando, tienes el terminal abierto. Tener que cambiar de contexto a una web rompe el flow.

Por qué importa: Los developers pasan el 80% del tiempo en terminal. Ver el último trace con treval trace last o hacer treval trace list --agent SupportAgent es más rápido que abrir un navegador.

Por qué treval puede hacerlo: Es una herramienta CLI desde el diseño — no un dashboard web con CLI añadido después. La terminal es el ciudadano de primera clase.

🗄️ SQLite local — cero infraestructura

Problema: Langfuse necesita Postgres + S3 (o cloud). AgentOps necesita su cloud. Phoenix necesita un servidor. DeepEval guarda resultados en JSON/cloud. Para un proyecto pequeño o un experimento rápido, montar infraestructura es una barrera.

Por qué importa: Si estás prototipando un agente, lo último que quieres es configurar una base de datos. SQLite corre en el mismo proceso, sin servidor, sin config.

Por qué treval puede hacerlo: SQLite es perfecto para el alcance de treval (datos estructurados de traces, sin concurrencia masiva). Y si el usuario necesita compartir datos, SQLite es un solo archivo — fácil de enviar, versionar, mover.

📡 Compatibilidad OTEL — sin vendor lock-in

Problema: AgentOps y DeepEval usan protocolos propietarios. Si mañana quieres cambiar de herramienta, pierdes todos los datos históricos.

Por qué importa: Los datos de observabilidad son inversión a largo plazo. Como usar OTEL es como usar HTTP — no piensas en el protocolo, solo funciona.

Por qué treval puede hacerlo: OTEL es el estándar CNCF. treval graba en SQLite usando el modelo de datos de OTEL. El usuario puede apuntar un exportador OTEL a Langfuse, Phoenix o Grafana y los mismos datos funcionan en todos lados.

🔄 Replay interactivo — modificar y re-ejecutar

Problema: Ninguna herramienta existente permite coger un trace de producción y re-ejecutarlo cambiando parámetros. Langfuse tiene "playground" pero solo para prompts, no para traces completos.

Por qué importa: Si un agente falló en producción, quieres poder replicar el fallo localmente, cambiar el prompt o el modelo, y ver si se soluciona — sin tener que escribir un script de test.

Por qué treval puede hacerlo: Como los traces están en SQLite local, el CLI puede leer un trace, preguntar "qué quieres cambiar" y re-ejecutar contra el mismo LLM. Es el debugger de agentes que no existe.

🧪 Testing nativo — evalúa agentes como evalúas código

Problema: DeepEval tiene CLI de testing pero no tracing. Langfuse tiene tracing pero no CLI de testing. Para tener ambos necesitas dos herramientas.

Por qué importa: El ciclo natural del desarrollo de agentes es: código → trace → debug → eval → código. Tener que saltar entre herramientas rompe el ciclo.

Por qué treval puede hacerlo: Como treval tiene tracing y evaluación en la misma codebase, un test puede tirar de un trace grabado, evaluarlo con métricas, y reportar — todo desde treval test run.

🎨 Decoradores con tipo — semántica de agente, no genérica

Problema: Langfuse usa @observe() para todo. No distingue entre una tool call, un paso de agente, o una operación interna. AgentOps sí lo hace, pero no es OTEL.

Por qué importa: Cuando miras un trace, no es lo mismo ver "tool call a search_web" que "LLM call a GPT-4". Los tipos de span correctos permiten filtrar, agregar y visualizar con precisión.

Por qué treval puede hacerlo: La jerarquía de AgentOps (SESSION → AGENT → OPERATION → TOOL) es el modelo correcto, pero treval lo implementa sobre OTEL + OpenInference, sin vendor lock-in. Lo mejor de ambos mundos.

5. Decisión confirmada — Stack técnico

10 junio 2026

La arquitectura de treval se basa en la combinación de 3 capas:

🎯 Decoradores API principal para el desarrollador. @treval.tool, @treval.agent, @treval.trace para instrumentar lógica de negocio sin fricción. ⚡ Auto-instrumentación Captura automática de LLMs (OpenAI, Anthropic) y HTTP — una sola línea (treval.instrument()) y zero cambios en código existente. 📊 OpenTelemetry (OTEL) Capa de transporte basada en estándar abierto CNCF. Garantiza: - Sin vendor lock-in (exportable a Langfuse, Grafana, Phoenix…) - Atributos semánticos LLM vía OpenInference - Compatibilidad con todo el ecosistema de observabilidad ┌─────────────────────────────────────────────────────────┐ │ ⚡ Auto-instrumentación → captura lo genérico │ │ + │ │ 🎯 Decoradores → captura lo específico │ │ ↓ │ │ 📊 Todo en formato OTEL → cualquier backend │ └─────────────────────────────────────────────────────────┘ Esta combinación cubre el 100% del pipeline de un agente con mínima fricción para el desarrollador.
